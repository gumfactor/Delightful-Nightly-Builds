# Future Features — worklog: Cross-Agent Project Activity Workstreams

> Ideas for extending this build.

---

## Quick Wins (under 1 hour to add)

1. **`worklog why` with workstream filter** — Accept `--workstream WS_ID` flag on `why` to scope decision search to one workstream. Currently `why` searches all decisions regardless of workstream. With a large ledger, this becomes noisy fast.

2. **`worklog sync --repo PATH`** — Allow syncing a repo that is not the current working directory. Right now sync always resolves from `cwd`. Trivial argparse + `ProjectContext.resolve(cwd=args.repo)` change.

3. **GITHUB_TOKEN auto-detection fallback** — When no env var, check `gh auth token` (GitHub CLI) as a fallback before giving up. One extra subprocess call, already available in most dev environments.

4. **Checkpoint validation dry-run** — `worklog checkpoint --file FILE --dry-run` that parses and reports what would be ingested without writing to the DB. Useful for checking checkpoint YAML syntax before committing a hook.

5. **`worklog events --since yesterday`** — Accept English time strings ("yesterday", "last week", "2 days ago") via `dateutil.relativedelta` or a simple parser, instead of requiring integer day counts.

## Medium Effort (roughly one nightly build session)

6. **Claude Code Hook integration** — Ship a ready-to-install Claude Code Hook that runs `worklog checkpoint --from-stdin` at session end, reading the Claude Code session context from the hook environment. Include an install script (`install-hook.sh`) that writes the hook definition to `.claude/settings.json`. This removes the last manual step.

7. **Workstream rename and merge commands** — `worklog workstreams rename WS_ID "New title"` and `worklog workstreams merge WS_A WS_B` with confirmation prompt. Currently workstream IDs are deterministic but titles are generated; this lets the user clean them up for clarity. Store user overrides in `.worklog.json`.

8. **`worklog resume --format markdown`** — Output the resume brief as a structured markdown document suitable for pasting directly into a GitHub issue, PR description, or Claude session starter. Currently outputs as plain prose.

9. **GitHub CI status collector** — Extend the GitHub collector to fetch check runs and workflow run results per PR/commit. Currently PRs are collected but their CI status is not. This lets `standup` report "blocked — CI red on PR #42" rather than just listing open PRs.

10. **Cross-project workstream view** — `worklog workstreams --all-projects` that queries all project DBs in `~/.worklog/` and shows activity across repos. Particularly useful for the user's parallel project work (Canada List, lab, nightly builds).

## Ambitious Extensions (multi-session effort)

11. **MCP server packaging** — Expose `sync`, `standup`, `resume`, `why`, and `workstreams` as MCP tool calls. This makes worklog callable from any Claude session without a terminal, turning it into a persistent project memory service across all AI surfaces. The SQLite ledger already supports concurrent readers; WAL mode is already configured.

12. **Local evidence graph visualization** — A lightweight single-file HTML dashboard (generated by `worklog export --html`) that renders workstreams as an interactive timeline with evidence links. Each event is a node; correlation signals are edges. Useful for auditing correlation decisions and understanding project history visually.

---

## Possible Integration Points

- **Git Standup Reporter** (builds/2026-06-07): worklog's standup view supersedes standup reporter for repos with multiple workstreams. The standup reporter's TOML config could be auto-generated from worklog's project detection.
- **Morning Briefing** (backlog ID 5): A future Morning Briefing routine should call `worklog standup --since 1` rather than running a separate git log command. worklog's structured output can feed directly into the briefing template.
- **AI Session Context Bridge** (builds/2026-06-06): worklog replaces the manual context capture that made AI Session Context Bridge low-rated. Future Claude Code hooks can write to worklog automatically, eliminating manual discipline.

## Known Limitations to Address

| Limitation | Suggested Fix |
|------------|---------------|
| Workstream correlation is stateless — recorrelates everything on every sync, which could reassign workstreams unexpectedly if new events are added | Add a `locked` flag to workstream records; locked workstreams are not reassigned by correlation |
| `why` search is exact substring match — misses plurals, synonyms, and related terms | Add SQLite FTS5 full-text index on decisions.summary and decisions.reason |
| Checkpoint YAML alternatives with `:` in text are parsed as dicts by PyYAML (YAML gotcha) | Document this in Manual.md; validate that all items in the alternatives list are strings and emit a clear error |
| GitHub collector fetches only 30–50 items per endpoint; for active repos with many PRs, older events are missed | Add pagination support with configurable max-pages limit |
| DB path uses home directory (`~/.worklog/`) — not ideal in containerized or CI environments | Support `WORKLOG_DB_DIR` env var to override the DB location |
